---
title: Publishing Errors & Retries
---

 <p>When theStacc publishes a blog to your website, it talks to your platform over the internet - WordPress, Ghost, Webflow, Zepio, Shopify, or your own webhook. Most of the time this just works. When it doesn't, theStacc tries to tell the difference between a problem that will fix itself (like your site being briefly busy) and a problem that needs you (like an expired password), and it handles each one differently.</p>
<p>This page explains what happens behind the scenes, what the error messages mean, and how to get a stuck blog published.</p>
<h2 id="two-kinds-of-problems">Two kinds of problems<a aria-label="Copy link to Two kinds of problems" class="heading-anchor" href="#two-kinds-of-problems">#</a></h2>
<p>Every publish failure falls into one of two buckets, and theStacc reacts differently to each.</p>
<h3 id="temporary-problems-thestacc-retries-for-you">Temporary problems - theStacc retries for you<a aria-label="Copy link to Temporary problems - theStacc retries for you" class="heading-anchor" href="#temporary-problems-thestacc-retries-for-you">#</a></h3>
<p>Some failures are not your fault and usually clear up on their own:</p>
<ul><li>Your platform returns a <strong>server error</strong> (a 5xx status)</li><li>Your platform says <strong>"too many requests, slow down"</strong> (a 429 rate limit)</li><li>The connection <strong>times out</strong> or drops mid-request (a network blip)</li></ul>
<p>When theStacc sees one of these, it does not give up. It waits a moment and tries again, automatically, <strong>up to 5 attempts</strong>. The wait gets longer each time so it doesn't hammer a struggling site - roughly 1 minute, then 2, then 4, then 8, then 16 minutes (the full retry window is about 30 minutes). You don't have to do anything; if one of the retries succeeds, the blog publishes and you'll see it go live.</p>
<p>If all 5 attempts fail, theStacc stops retrying and marks the blog <strong>publish failed</strong> so you can take a look.</p>
<h3 id="permanent-problems-thestacc-stops-and-tells-you">Permanent problems - theStacc stops and tells you<a aria-label="Copy link to Permanent problems - theStacc stops and tells you" class="heading-anchor" href="#permanent-problems-thestacc-stops-and-tells-you">#</a></h3>
<p>Other failures won't be fixed by trying again, so theStacc doesn't waste time retrying:</p>
<ul><li>A <strong>login or permission problem</strong> (your platform returns a 401 or 403)</li><li>A <strong>bad request</strong> your platform rejects (most 4xx errors, like a validation error or a slug conflict)</li><li>A <strong>security block</strong> - if a publishing URL points at a private or internal address instead of a public website, theStacc refuses to send your content there</li></ul>
<p>These stop immediately and the blog is marked <strong>publish failed</strong> with a message explaining what to fix. Retrying the exact same thing would just fail the exact same way, so theStacc waits for you to correct the underlying issue and republish.</p>
<h2 id="why-most-failures-don-t-set-off-alarms">Why most failures don't set off alarms<a aria-label="Copy link to Why most failures don't set off alarms" class="heading-anchor" href="#why-most-failures-don-t-set-off-alarms">#</a></h2>
<p>The vast majority of publish failures are configuration issues - an expired token, a platform that isn't connected, a password that needs updating, a slug that already exists. theStacc treats these as <strong>warnings</strong>, not emergencies. They're expected, they're yours to fix, and they show up clearly on the affected blog so you know exactly what to do.</p>
<p>theStacc only treats a failure as an internal emergency when it looks like a genuine bug on our side - never for a normal credential or configuration error. In plain terms: an expired WordPress password is your cue to reconnect, not a reason for our team to get paged at 3 a.m.</p>
<h2 id="new-post-vs-republish">New post vs. republish<a aria-label="Copy link to New post vs. republish" class="heading-anchor" href="#new-post-vs-republish">#</a></h2>
<p>theStacc remembers where each blog lives on your platform, which changes how publishing behaves:</p>
<ul><li><strong>First publish</strong> creates a brand-new post on your platform and records its ID. This is sent as a <code>blog.published</code> event to webhooks.</li><li><strong>Republish</strong> updates the existing post in place, using the stored ID - it does not create a second copy. This is sent as a <code>blog.updated</code> event to webhooks.</li></ul>
<p>So if you edit a blog and publish it again, your readers see the same post updated, not a duplicate. (For platforms that can't update an existing post, like Zepio, republish isn't possible - see the Zepio note below.)</p>
<h2 id="how-thestacc-avoids-duplicate-posts-on-retry">How theStacc avoids duplicate posts on retry<a aria-label="Copy link to How theStacc avoids duplicate posts on retry" class="heading-anchor" href="#how-thestacc-avoids-duplicate-posts-on-retry">#</a></h2>
<p>There's one tricky case worth knowing about. Sometimes theStacc sends your post, your platform actually creates it, but the "success" reply gets lost on the way back. theStacc thinks it failed and schedules a retry - which could create a second copy of the same post.</p>
<p>To prevent that, before retrying a first-time publish, theStacc first checks your platform to see whether the previous attempt already created the post. It matches on the post's <strong>slug</strong> plus a <strong>matching title created within roughly the last 35 minutes</strong> (comfortably longer than the whole retry window). If it finds that the post is already there, theStacc treats the publish as done instead of creating a duplicate.</p>
<p>This check is deliberately cautious: it only claims a match when the title lines up and the timing fits, so an unrelated post you happened to publish under the same slug is never mistaken for theStacc's work.</p>
<h2 id="what-the-per-platform-messages-mean">What the per-platform messages mean<a aria-label="Copy link to What the per-platform messages mean" class="heading-anchor" href="#what-the-per-platform-messages-mean">#</a></h2>
<p>When a publish fails permanently, the message on the blog is tailored to your platform so you know precisely what to fix.</p>
<h3 id="wordpress">WordPress<a aria-label="Copy link to WordPress" class="heading-anchor" href="#wordpress">#</a></h3>
<ul><li><strong>"It appears to be a regular login password, not a WordPress Application Password"</strong> - You entered your normal WordPress login password. WordPress needs a dedicated <strong>Application Password</strong> instead. Generate one in <strong>WordPress Admin &gt; Users &gt; Profile &gt; Application Passwords</strong> and update it in your integration settings.</li><li><strong>"Authentication failed (401 Unauthorized)"</strong> - Your username or Application Password is wrong or has been revoked. Double-check the username and regenerate the Application Password.</li><li><strong>"403 Forbidden"</strong> - The connected WordPress user doesn't have permission to create posts. Make sure that user has an Administrator, Editor, or Author role.</li><li><strong>"WordPress.com token expired. Please reconnect your WordPress.com account."</strong> - For WordPress.com (OAuth) connections, your access has lapsed. Reconnect the account from <strong>Content SEO &gt; Settings &gt; Publishing</strong>.</li></ul>
<h3 id="ghost">Ghost<a aria-label="Copy link to Ghost" class="heading-anchor" href="#ghost">#</a></h3>
<ul><li><strong>"Ghost authentication failed. Please check your Admin API key."</strong> - The Admin API key is wrong, malformed, or has been removed in Ghost. Recreate it in <strong>Ghost Admin &gt; Settings &gt; Integrations</strong> and paste the new key in.</li><li><strong>"Ghost validation error"</strong> - Ghost rejected something in the post itself. The message includes Ghost's own explanation.</li></ul>
<h3 id="zepio">Zepio<a aria-label="Copy link to Zepio" class="heading-anchor" href="#zepio">#</a></h3>
<ul><li><strong>"Your Zepio site is rate-limiting requests right now."</strong> - Zepio is temporarily throttling. theStacc retries automatically; no action needed unless it persists.</li><li><strong>"Zepio Developer API not found at this URL."</strong> - The site URL is wrong, or the Developer API isn't enabled on your Zepio plan. Confirm the URL (for example <code>https://mybusiness.com</code>) and that the Developer API is turned on.</li><li><strong>"Invalid Developer API key"</strong> - The key was rejected. Check it under <strong>Zepio Admin &gt; Settings &gt; Developer API</strong>.</li><li><strong>Republish not supported</strong> - Zepio's Developer API can only create posts, not update them. To change a published Zepio post, edit it directly in your Zepio dashboard, or delete it there and publish again.</li></ul>
<h3 id="webflow">Webflow<a aria-label="Copy link to Webflow" class="heading-anchor" href="#webflow">#</a></h3>
<ul><li><strong>"Webflow authentication failed. Please re-verify your API token." / "Invalid API token"</strong> - Your Webflow API token is wrong or revoked. Generate a new one and reconnect.</li><li><strong>"Webflow site has never been published."</strong> - Webflow won't accept content until the site has been published at least once. Open the Webflow Designer, click <strong>Publish</strong> once, then republish from theStacc.</li><li><strong>"Item with this slug already exists in Webflow."</strong> - A different item already uses that slug. Change the blog's slug and republish. (On a first publish, theStacc tries adding a numbered suffix automatically before showing this.)</li><li><strong>Access denied / 403</strong> - The API token doesn't have the right permissions, or the Webflow plan doesn't include the CMS. Make sure the token has <strong>CMS: Read and write</strong> and <strong>Sites: Read and publish</strong> access, and that your Webflow plan includes the CMS.</li><li><strong>"Webflow validation error"</strong> - Webflow rejected one or more fields; the message lists which field and why.</li></ul>
<h3 id="custom-webhook">Custom webhook<a aria-label="Copy link to Custom webhook" class="heading-anchor" href="#custom-webhook">#</a></h3>
<ul><li><strong>"Webhook receivers must respond with 2xx directly" (redirect rejected)</strong> - Your endpoint replied with a 3xx redirect. For security, theStacc does not follow redirects on publish - your receiver must accept the request and respond with a 2xx status itself.</li><li><strong>"Webhook URL rejected"</strong> - The URL points at an internal or private address. Publishing URLs must point at a public host.</li><li><strong>HTTPS required</strong> - Webhook endpoints must use <strong>HTTPS</strong>, not plain HTTP.</li><li><strong>No live URL returned</strong> - This isn't an error. Your endpoint accepted the post but didn't return a URL, so theStacc marks it published without a clickable live link. That's expected for fire-and-forget receivers.</li></ul>
<h2 id="where-to-see-what-went-wrong">Where to see what went wrong<a aria-label="Copy link to Where to see what went wrong" class="heading-anchor" href="#where-to-see-what-went-wrong">#</a></h2>
<p>Two places tell you the story:</p>
<ol><li><strong>The blog itself.</strong> A blog that failed to publish shows a <strong>publish failed</strong> state with the most recent error message - the same per-platform text described above. This is the fastest way to see why a specific post didn't go live.</li><li><strong>The integration's test history.</strong> Under <strong>Content SEO &gt; Settings &gt; Publishing</strong>, each connected platform records when it was <strong>last tested</strong>, whether that <strong>last test succeeded</strong>, and the <strong>connection error</strong> from the most recent failure. If several blogs are failing at once, this is usually the quickest way to confirm it's a credential or connection problem rather than something about one specific post.</li></ol>
<p>Whenever a publish hits an authentication failure, theStacc also flags the integration's last test as unsuccessful - so a green "last test succeeded" is a good sign your credentials are still valid.</p>
<h2 id="how-to-recover">How to recover<a aria-label="Copy link to How to recover" class="heading-anchor" href="#how-to-recover">#</a></h2>
<p>For a blog stuck in <strong>publish failed</strong>, the fix is almost always the same three steps:</p>
<ol><li><strong>Fix the underlying issue.</strong> Read the error on the blog and follow it - reconnect WordPress.com, paste a fresh Ghost Admin API key, generate a WordPress Application Password, change a conflicting slug, publish your Webflow site once, and so on.</li><li><strong>Re-test the connection.</strong> Go to <strong>Content SEO &gt; Settings &gt; Publishing</strong>, open the affected integration, and run the connection test. A successful test confirms the credentials are good before you spend a publish on it.</li><li><strong>Republish.</strong> Open the blog and click <strong>Republish</strong>. Since theStacc remembers the post, a republish updates the existing post rather than creating a duplicate.</li></ol>
<p>If the integration was turned off or disconnected, re-enable or reconnect it first - theStacc will not auto-publish through an inactive integration.</p>

title: Connect Platforms
---

 <p>theStacc publishes your generated blog posts straight to your website. Connect one of the supported destinations and, depending on your publishing mode, new posts go live (or land as drafts) automatically. Manage every connection under <strong>Content SEO &gt; Settings &gt; Publishing</strong>, in the <strong>Destination</strong> section.</p>
<p>Supported destinations:</p>
<ul><li><strong>WordPress.com</strong> (hosted WordPress)</li><li><strong>WordPress.org</strong> (self-hosted WordPress)</li><li><strong>Custom Webhook</strong> (any system that accepts an HTTP POST)</li><li><strong>Ghost</strong></li><li><strong>Webflow</strong></li><li><strong>Shopify</strong> (Beta)</li><li><strong>Zepio</strong></li></ul>
<h2 id="two-rules-that-apply-to-every-destination">Two rules that apply to every destination<a aria-label="Copy link to Two rules that apply to every destination" class="heading-anchor" href="#two-rules-that-apply-to-every-destination">#</a></h2>
<p><strong>Only one destination is active at a time.</strong> A project publishes to a single place. When you turn on a new destination, theStacc automatically turns the previous one off — your old connection settings are kept, so you can switch back later without re-entering anything. The Publishing page says it plainly under Destination: *"One destination is active at a time."*</p>
<p><strong>You need to be a project admin to connect, switch, or test a destination.</strong> Connecting a platform, toggling it on or off, editing its settings, and running a test are all admin-only actions. If you're a member without admin rights, the controls are visible but disabled. (Ask your project owner or an admin to make the change, or to grant you admin on the project.)</p>
<p>With those out of the way, here's each destination.</p>
<h2 id="wordpress-com">WordPress.com<a aria-label="Copy link to WordPress.com" class="heading-anchor" href="#wordpress-com">#</a></h2>
<p>Hosted WordPress sites. Recommended for WordPress.com users — no passwords to copy, just a secure sign-in.</p>
<ol><li>Click <strong>Connect WordPress.com</strong>.</li><li>You'll be redirected to WordPress.com to securely authorize access (OAuth).</li><li>Sign in and grant theStacc access.</li><li>Select your target site if your account has more than one.</li></ol>
<p>Each published blog includes the title, body content (HTML), featured image, categories, tags, SEO meta data (Yoast/RankMath compatible), and post slug.</p>
<p>If the connection ever stops working, you'll see *"WordPress.com token expired or invalid. Please reconnect your account."* — click Connect again to re-authorize.</p>
<h2 id="wordpress-org">WordPress.org<a aria-label="Copy link to WordPress.org" class="heading-anchor" href="#wordpress-org">#</a></h2>
<p>Self-hosted WordPress sites. Connect with an <strong>Application Password</strong> — a WordPress feature made specifically for apps like theStacc, so you never share your real login password.</p>
<ol><li>In your WordPress admin, go to <strong>Users &gt; Profile &gt; Application Passwords</strong>.</li><li>Create a new password and name it <code>theStacc</code>.</li><li>Copy the generated password.</li><li>In theStacc, click <strong>Connect WordPress.org (self-hosted)</strong>, then enter your <strong>site URL</strong>, your <strong>WordPress username</strong> (your login username, not your display name), and the <strong>application password</strong>.</li><li>theStacc tests the connection as you save. If it succeeds, you're connected; if not, you'll get a specific reason (see the error table below).</li></ol>
<p>Requires WordPress 5.6+ with the REST API enabled (on by default). HTTPS is recommended.</p>
<p><strong>About the application password field.</strong> WordPress shows application passwords as groups separated by spaces (for example <code>abcd 1234 efgh 5678 ijkl 9012</code>). You can paste it with or without the spaces — theStacc strips them for you. After stripping spaces it must be <strong>8 to 200 characters</strong> and contain <strong>only letters and numbers</strong>. If it contains special characters, theStacc rejects it on the spot, because that almost always means you pasted your normal WordPress login password by mistake. Generate a real Application Password from <strong>Users &gt; Profile &gt; Application Passwords</strong> instead.</p>
<p><strong>Publishing permission.</strong> The WordPress user must be an <strong>Administrator, Editor, or Author</strong> — those are the roles that can create posts. If the user can sign in but can't publish, theStacc tells you the exact role it found and which roles are allowed.</p>
<h2 id="custom-webhook">Custom Webhook<a aria-label="Copy link to Custom Webhook" class="heading-anchor" href="#custom-webhook">#</a></h2>
<p>Send blog posts to any platform via HTTP. Works with any system that accepts HTTP POST requests — a custom CMS, a static-site build pipeline, an internal tool, or a no-code automation.</p>
<blockquote><p><strong>Building the receiver?</strong> See the <a href="/docs/developers/webhook-reference/">Webhook Reference</a> for the full developer guide — handshake, signing, every event, every field, response contract, database schemas, and ready-to-paste receiver examples in 6 languages.</p></blockquote>
<ol><li>Click <strong>Add Webhook</strong>.</li><li>Enter your <strong>Webhook URL</strong>.</li><li>Add an optional <strong>Secret Key</strong> — when set, theStacc signs every request so your receiver can verify it really came from theStacc. The secret must be <strong>at least 16 characters</strong> (and at most 256).</li><li>Add <strong>custom headers</strong> (optional) — for authentication tokens, API keys, or routing.</li><li>Click <strong>Test Connection</strong> to verify the URL is reachable.</li><li>Click <strong>Sample Payload</strong> to send the real publish-shape JSON and validate your parser.</li></ol>
<h3 id="two-events-your-receiver-must-handle">Two events your receiver must handle<a aria-label="Copy link to Two events your receiver must handle" class="heading-anchor" href="#two-events-your-receiver-must-handle">#</a></h3>
<p>theStacc sends a different payload shape for each button. Branch on the <code>event</code> field — your receiver should handle both, or it will fail Test Connection even when the publish path works.</p>

<h3 id="verify-webhook-signatures-recommended">Verify webhook signatures (recommended)<a aria-label="Copy link to Verify webhook signatures (recommended)" class="heading-anchor" href="#verify-webhook-signatures-recommended">#</a></h3>
<p>When you configure a webhook secret, theStacc signs every request with HMAC-SHA256 and sends the hex digest in the <code>X-Webhook-Signature</code> header. Verify it on every request — without verification, anyone who guesses your endpoint URL can post fake blogs to your CMS.</p>
<p><strong>Critical:</strong> the signature is computed over the <strong>raw request body</strong> that theStacc sent, which is JSON serialized in compact form (<code>json.dumps(payload, separators=(',', ':'))</code> — no whitespace between keys and values). If you re-serialize the parsed JSON before hashing, the byte-for-byte representation will differ and the hashes won't match. <strong>Always hash the raw bytes you receive over the wire.</strong></p>

<h3 id="operational-details">Operational details<a aria-label="Copy link to Operational details" class="heading-anchor" href="#operational-details">#</a></h3>
<ul><li><strong>Timeout:</strong> theStacc waits up to <strong>15 seconds</strong> for a 2xx response. Slow CMS writes (image uploads, search indexing) will time out — return 2xx fast and do heavy work async.</li><li><strong>No automatic retries.</strong> A 5xx or timeout fails the publish in theStacc and the user sees a "Failed to publish" toast. If your endpoint is occasionally slow, queue the actual CMS write internally and return 2xx immediately.</li><li><strong>Redirects rejected.</strong> A 3xx response is treated as failure (anti-SSRF guardrail; a public webhook receiver that 302s to an internal address would otherwise bypass URL safety checks).</li><li><strong>HTTPS only.</strong> <code>http://</code> URLs and internal/private IP ranges are rejected at save time.</li><li><strong>Idempotency.</strong> Use <code>blog_id</code> as a dedupe key. If theStacc ever resends (manual user retry), you'll see the same <code>blog_id</code>.</li><li><strong>Test before writing receiver code.</strong> Point your webhook at <a href="https://webhook.site">webhook.site</a> or <a href="https://requestbin.com">requestbin.com</a> first to inspect the actual request body and headers, then build your receiver against the real shape.</li></ul>
<h3 id="best-practices">Best practices<a aria-label="Copy link to Best practices" class="heading-anchor" href="#best-practices">#</a></h3>
<ul><li>Use HTTPS — theStacc rejects <code>http://</code> and internal addresses.</li><li>Return a 2xx status code. Redirects (3xx) are rejected.</li><li>Verify the <code>X-Webhook-Signature</code> header on every request when a secret is set.</li><li>Idempotency — use <code>blog_id</code> as a dedupe key in case of retries.</li><li>Set up error alerting on your endpoint so silent 4xx/5xx responses don't go unnoticed.</li></ul>
<h2 id="ghost">Ghost<a aria-label="Copy link to Ghost" class="heading-anchor" href="#ghost">#</a></h2>
<p>Publish directly to your Ghost blog via the Ghost Admin API.</p>
<ol><li>Click <strong>Connect Ghost</strong>.</li><li>Enter your <strong>Ghost site URL</strong> (for example <code>https://myblog.com</code>).</li><li>Enter your <strong>Ghost Admin API key</strong>. <p>Posts are created with full content, featured images, tags, and meta data.</p>

If the format is off, you'll see *"Invalid API key format. Expected format: {id}:{secret}"* or *"Invalid API key secret — must be a valid hex string."*</p>

<h3 id="changing-webflow-settings-later-no-token-re-entry">Changing Webflow settings later (no token re-entry)<a aria-label="Copy link to Changing Webflow settings later (no token re-entry)" class="heading-anchor" href="#changing-webflow-settings-later-no-token-re-entry">#</a></h3>
<p>Open the connected Webflow integration's <strong>Edit settings</strong> to change the publishing behavior, the selected site or collection, or your field mappings — without re-pasting the API token. For security, theStacc masks the saved token and never shows it back to you, so the edit form simply leaves it untouched unless you deliberately paste a new one.</p>
<p>If you do paste a new token, theStacc validates it against Webflow before saving. A typo or a revoked token is rejected with a clear message (*"Webflow API token is invalid or expired…"* or *"…lacks required scopes (sites:read, cms:read, cms:write)"*) so a bad token can never silently replace your working one.</p>

<h3 id="troubleshooting">Troubleshooting<a aria-label="Copy link to Troubleshooting" class="heading-anchor" href="#troubleshooting">#</a></h3>
<ul><li><strong>"Invalid API token"</strong> — token wrong, expired, or revoked. Generate a new one and update the integration.</li><li><strong>"No CMS collections found"</strong> — the site has no CMS collections. Create one in Designer first.</li><li><strong>"Item with this slug already exists"</strong> — a post with that slug was already published. Edit the slug or delete the existing item in Webflow.</li><li><strong>Post published but URL returns 404</strong> — most often the site has never been published from the Designer, or you're using a custom domain whose DNS hasn't propagated. Open Webflow Designer and click Publish at least once.</li><li><strong>"409 Site not published" on first publish</strong> — Webflow rejects live CMS publishes on un-published sites. theStacc automatically falls back to staged item creation. Publish your site once in Designer; future live publishes work directly.</li><li><strong>403 Forbidden when creating items</strong> — the workspace plan doesn't include CMS API access. Upgrade to a Workspace plan with CMS hosting.</li></ul>
<h3 id="what-gets-sent-to-webflow">What gets sent to Webflow<a aria-label="Copy link to What gets sent to Webflow" class="heading-anchor" href="#what-gets-sent-to-webflow">#</a></h3>
<p>theStacc sends <code>POST https://api.webflow.com/v2/collections/{id}/items/live</code> (or <code>/items</code> for staged) with your mapped fields plus the required <code>name</code> and <code>slug</code>. No source code, design changes, or custom integrations are needed on the Webflow side.</p>
<h2 id="shopify-beta">Shopify (Beta)<a aria-label="Copy link to Shopify (Beta)" class="heading-anchor" href="#shopify-beta">#</a></h2>
<p>Publish blog posts to your Shopify store's built-in blog. Shopify support is currently <strong>Beta</strong>.</p>
<ol><li>Click <strong>Connect Shopify</strong>.</li><li>Enter your store domain — you can type just the handle (<code>your-store</code>), the full <code>your-store.myshopify.com</code>, or even a pasted store URL. theStacc normalizes whatever you enter to the canonical <code>your-store.myshopify.com</code> form. (You'll find this in your Shopify admin URL.)</li><li>You're redirected to Shopify to approve access (OAuth). theStacc never sees your Shopify password.</li><li>After you approve, theStacc connects, finds the blog to publish to, and makes Shopify the active destination. If your store has no blog yet, theStacc creates one named <strong>News</strong> so there's somewhere for posts to land.</li></ol>
<p>Posts are published as Shopify blog articles with content, images, and tags, via Shopify's GraphQL Admin API. As with every destination, connecting Shopify deactivates whatever was active before — one destination at a time per project.</p>
<p>If the connection is lost (for example you uninstall the app on Shopify's side), you'll see *"Shopify access token is invalid or expired. Please reconnect."* — click Connect Shopify again to re-authorize.</p>
<h2 id="zepio">Zepio<a aria-label="Copy link to Zepio" class="heading-anchor" href="#zepio">#</a></h2>
<p>Publish blog posts directly to your <a href="https://zepio.io">Zepio</a> website using Zepio's Developer API.</p>
<p><strong>What you'll need:</strong></p>
<ul><li>Your Zepio website URL</li><li>A Zepio <strong>Developer API key</strong> with the <strong>"Create blog posts"</strong> permission</li></ul>
<p><strong>Steps:</strong></p>
<ol><li>In Zepio, go to <strong>Zepio Admin → Settings → Developer API</strong> and create an API key with the <strong>"Create blog posts"</strong> permission.</li><li>In theStacc, click <strong>Connect Zepio</strong>.</li><li>Enter your <strong>Zepio Website URL</strong> (for example <code>https://mybusiness.com</code>).</li><li>Paste your <strong>Developer API key</strong>.</li><li>Click <strong>Save Integration</strong>. theStacc verifies the key with Zepio before saving — if the key is wrong or lacks permission, it tells you immediately rather than failing silently at publish time.</li></ol>
<p>The key is <strong>10 to 500 characters</strong>. Posts are created with title, content, featured image, excerpt, and SEO meta data.</p>
<p><strong>Editing later — rotate the key or change the URL.</strong> Once Zepio is connected, an <strong>Edit</strong> button (tooltip: *"Update site URL or rotate the API key"*) lets you change the site URL or paste a new Developer API key without disconnecting and reconnecting from scratch. For security theStacc never shows the saved key back to you, so re-enter it (or paste a fresh one) when you save changes.</p>
<p><strong>Note on editing published posts.</strong> Zepio's Developer API can create a post but has no "update" endpoint, so once a post is published to Zepio you edit it from your <strong>Zepio dashboard</strong>, not from theStacc. Re-publishing the same post from theStacc isn't supported for Zepio for the same reason.</p>
<h2 id="status-and-diagnostics">Status and diagnostics<a aria-label="Copy link to Status and diagnostics" class="heading-anchor" href="#status-and-diagnostics">#</a></h2>
<p>Every connected integration carries connection-health information theStacc updates each time it tests or publishes:</p>
<ul><li><strong>Last tested</strong> — when the connection was last checked.</li><li><strong>Last test success</strong> — whether that most recent check passed or failed.</li><li><strong>Connection error</strong> — the specific reason the last check failed (shown when you connect or test). This is the message to read first when something isn't working.</li></ul>
<p>Use the <strong>Test Connection</strong> button (admin only) any time to re-run the check on demand. WordPress, Ghost, and Zepio are also tested automatically the moment you connect them, so a bad credential is caught up front instead of at the next scheduled publish.</p>

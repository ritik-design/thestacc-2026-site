title: Publishing

 <p>Once content is generated, theStacc handles publishing to your website. Configure your publishing mode and connections under <strong>Content SEO &gt; Settings &gt; Publishing</strong>.</p>
<figure class="docs-figure"><img alt="How a generated post reaches your site: your publishing mode decides whether it auto-publishes, waits for approval, or stays a draft" loading="lazy" src="/images/docs/publishing-flow.svg"><figcaption>How a generated post reaches your site: your publishing mode decides whether it auto-publishes, waits for approval, or stays a draft</figcaption></img></figure>
<h2 id="publishing-modes">Publishing modes<a aria-label="Copy link to Publishing modes" class="heading-anchor" href="#publishing-modes">#</a></h2>
<p>Every project has a publishing mode that decides what happens to a blog after it is generated and SEO-scored. Change it any time under <strong>Content SEO &gt; Settings &gt; Publishing</strong>.</p>
<h3 id="no-auto-publish">No auto publish<a aria-label="Copy link to No auto publish" class="heading-anchor" href="#no-auto-publish">#</a></h3>
<p>Blogs are saved as drafts. You manually review, edit, and publish each one when you are ready. Best for maximum control over every piece of content.</p>
<h3 id="requires-approval">Requires approval<a aria-label="Copy link to Requires approval" class="heading-anchor" href="#requires-approval">#</a></h3>
<p>Blogs enter a "Pending Review" state. You get a notification, review the post, make edits if needed, then approve or reject. Rejected posts can be regenerated. This is the best balance of control and convenience.</p>
<h3 id="auto-publish">Auto publish<a aria-label="Copy link to Auto publish" class="heading-anchor" href="#auto-publish">#</a></h3>
<p>Blogs publish automatically after generation and SEO scoring, with no review step. You can still unpublish or edit after a post goes live. Auto publish requires an active publishing destination. Best for hands-off operation.</p>
<p>For a deeper walkthrough of autopilot, daily auto-generation, and how the cron pipeline schedules and spaces out publishes, see <a href="/docs/content-seo/publishing-modes/">Publishing Modes &amp; Autopilot</a>.</p>
<h2 id="supported-platforms">Supported platforms<a aria-label="Copy link to Supported platforms" class="heading-anchor" href="#supported-platforms">#</a></h2>
<p>theStacc publishes to all of these. Connect one as your active destination under <strong>Content SEO &gt; Settings &gt; Publishing</strong>:</p>
<ul><li><strong>WordPress.com</strong> - hosted WordPress sites, connected with a secure OAuth sign-in.</li><li><strong>WordPress.org (self-hosted)</strong> - your own WordPress install, connected with a WordPress Application Password.</li><li><strong>Ghost</strong> - connected with your Ghost site URL and an Admin API key.</li><li><strong>Webflow</strong> - publishes blog posts as items in a Webflow CMS collection, connected with a Webflow API token.</li><li><strong>Zepio</strong> - connected with a Zepio Developer API key.</li><li><strong>Custom Webhook</strong> - sends each publish as JSON to any HTTP endpoint you control, for sites and CMSes not directly supported.</li><li><strong>Shopify (beta)</strong> - publishes to your Shopify store's built-in blog, connected with a secure OAuth sign-in.</li></ul>
<p>For step-by-step setup of every platform, including screenshots and field mapping, see <a href="/docs/content-seo/connect-platforms/">Connect Platforms</a>.</p>
<h3 id="one-active-destination-at-a-time">One active destination at a time<a aria-label="Copy link to One active destination at a time" class="heading-anchor" href="#one-active-destination-at-a-time">#</a></h3>
<p>A project publishes to exactly <strong>one</strong> destination at a time. Connecting a new platform automatically makes it the active one and deactivates whatever was active before, so a blog is never pushed to two places at once.</p>
<p>You can keep several connections saved on a project, including more than one of the same type (for example, two different webhook endpoints). Only the one marked active receives publishes. To switch destinations, activate a different connection from the list and the previously active one is turned off automatically.</p>
<p>Autopilot is strict about this: the auto-publish pipeline only ever uses the active connection. If you deactivate every connection, autopilot has nowhere to publish and the blog is held with a clear message asking you to re-enable a destination.</p>
<h3 id="who-can-change-connections">Who can change connections<a aria-label="Copy link to Who can change connections" class="heading-anchor" href="#who-can-change-connections">#</a></h3>
<p>Changing publishing connections (connecting, editing, activating, deactivating, testing, or disconnecting) requires <strong>editor or admin</strong> access to the project. On a project you own, any member of your workspace can make these changes; on a shared project, members with view-only access cannot.</p>
<h3 id="credentials-are-tested-before-they-are-saved">Credentials are tested before they are saved<a aria-label="Copy link to Credentials are tested before they are saved" class="heading-anchor" href="#credentials-are-tested-before-they-are-saved">#</a></h3>
<p>When you save a connection, theStacc tests the credentials against the platform first. If the test fails, nothing is saved and you get the exact reason (wrong URL, bad API key, a user role that cannot publish, and so on). There are no partial saves: a connection is only stored once it has proven it can reach your site.</p>
<p>The same test runs whenever you change a credential later, so a typo or a revoked token is caught at save time rather than at the next publish. You can also re-run the test any time from the connection's <strong>Test</strong> button.</p>
<h2 id="how-content-is-prepared-before-it-is-sent">How content is prepared before it is sent<a aria-label="Copy link to How content is prepared before it is sent" class="heading-anchor" href="#how-content-is-prepared-before-it-is-sent">#</a></h2>
<p>The blog you see in the editor is cleaned up at publish time so it renders correctly on your site no matter which platform you use:</p>
<ul><li><strong>The hero image is removed from the body.</strong> Your featured image is sent as a separate structured field that every platform renders at the top of the post. theStacc strips that same image out of the article body so it does not appear twice on the page.</li><li><strong>The excerpt is trimmed.</strong> A short, clean excerpt (capped at 256 characters and trimmed at a word boundary) is sent alongside the post, so platforms with excerpt length limits accept it without error.</li><li><strong>Internal link placeholders are resolved.</strong> While writing, the AI marks internal links as placeholders. At publish time these are matched to your configured internal links and turned into real URLs. Any placeholder with no match is removed and the visible link text is kept, so a broken link is never published. See <a href="/docs/content-seo/internal-linking/">Internal Linking</a> for how to set up the link targets.</li><li><strong>Relative links are made absolute.</strong> Call-to-action links entered as a path (like <code>/contact</code>) are rewritten to your full site URL (like <code>https://yoursite.com/contact</code>) so they work everywhere the post is shown, including the in-app preview.</li></ul>
<p>The duplicate H1 is also removed, since platforms render your title as the page heading.</p>
<h2 id="when-a-publish-fails">When a publish fails<a aria-label="Copy link to When a publish fails" class="heading-anchor" href="#when-a-publish-fails">#</a></h2>
<p>Not every publish succeeds on the first try, so theStacc tells the difference between a hiccup and a real problem.</p>
<h3 id="temporary-problems-are-retried">Temporary problems are retried<a aria-label="Copy link to Temporary problems are retried" class="heading-anchor" href="#temporary-problems-are-retried">#</a></h3>
<p>If a publish fails for a reason that is likely to clear up on its own (the platform returns a server error or a rate-limit, or the request times out), theStacc retries automatically with increasing delays: roughly 1, 2, 4, 8, then 16 minutes apart. It tries up to <strong>5</strong> times in total. If all 5 attempts fail, the blog is marked <strong>publish failed</strong> and you are shown the last error.</p>
<p>On a retry, theStacc first checks your site to see whether the earlier attempt actually landed (the response can be lost in transit even when the post was created). If it finds the matching post, it counts that as success instead of creating a duplicate.</p>
<h3 id="problems-you-need-to-fix-are-not-retried">Problems you need to fix are not retried<a aria-label="Copy link to Problems you need to fix are not retried" class="heading-anchor" href="#problems-you-need-to-fix-are-not-retried">#</a></h3>
<p>Some failures will never fix themselves, the most common being an expired token or an invalid credential (a 401 or 403 from your platform, or a URL theStacc cannot reach). These point at something only you can change, so theStacc does not waste retries on them. The blog is marked publish failed immediately with a message telling you exactly what to fix, such as reconnecting your account or generating a fresh Application Password.</p>
<p>Because these are expected user-configuration issues rather than bugs, they do not page the engineering on-call team. Once you have updated the connection, click <strong>Re-Publish</strong> on the blog to try again. For the full list of error messages and how to clear each one, see <a href="/docs/content-seo/publishing-errors/">Publishing Errors &amp; Retries</a>.</p>
<h2 id="custom-webhook-events">Custom webhook events<a aria-label="Copy link to Custom webhook events" class="heading-anchor" href="#custom-webhook-events">#</a></h2>
<p>If you publish through a custom webhook, your receiver gets a JSON request for each event. Branch on the <code>event</code> field:</p>
<div class="table-wrap"><table><thead><tr><th>Event</th><th>When it fires</th></tr></thead><tbody><tr><td><code>test.ping</code></td><td>The <strong>Test Connection</strong> button. A liveness check only, with no blog fields.</td></tr><tr><td><code>blog.published</code></td><td>The first successful publish of a blog (and the <strong>Sample Payload</strong> button, with a <code>preview-</code> prefixed <code>blog_id</code>). Carries the full blog payload.</td></tr><tr><td><code>blog.updated</code></td><td>A re-publish of a blog that was already published. Same payload shape as <code>blog.published</code>.</td></tr><tr><td><code>blog.unpublished</code></td><td>You unpublish a blog from theStacc.</td></tr><tr><td><code>blog.deleted</code></td><td>You delete a published blog from theStacc.</td></tr></tbody></table></div>
<p>Respond to publish events with a <code>2xx</code> status and a JSON body like <code>{"ok": true, "url": "&lt;live-post-url&gt;", "id": "&lt;your-cms-id&gt;"}</code> so theStacc can store the live URL and show a clickable <strong>View live post</strong> link. The <code>id</code> you return is remembered so later updates, unpublishes, and deletes can target the right record.</p>
<p>For the complete request and response contract, HMAC signature verification, timeouts, and ready-to-paste receiver examples, see <a href="/docs/content-seo/connect-platforms/">Connect Platforms - Custom Webhook</a> and the <a href="/docs/developers/webhook-reference/">Webhook Reference</a>.</p>
<h2 id="managing-connections">Managing connections<a aria-label="Copy link to Managing connections" class="heading-anchor" href="#managing-connections">#</a></h2>
<p>Each saved connection shows:</p>
<ul><li><strong>Status</strong> - active, inactive, or disconnected</li><li><strong>Last test result</strong> - whether the most recent connection test passed</li><li><strong>Test</strong> - re-run the credential check at any time</li><li><strong>Active toggle</strong> - make this the publishing destination (which deactivates the others)</li><li><strong>Disconnect</strong> - remove the connection entirely</li></ul>

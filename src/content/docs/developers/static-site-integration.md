title: Static Site Integration
---

 <p>This guide walks you through connecting theStacc to your static site. By the end, your site will automatically display AI-generated blog posts — fully rendered as static HTML for perfect SEO.</p>
<p>It uses the <a href="/docs/developers/public-blog-api/">Public Blog API</a>, which exposes your published posts as JSON. Your site fetches that JSON at build time and turns it into static HTML pages, so Google sees ordinary, fully rendered pages — it has no idea the content came from an API.</p>
<h2 id="prerequisites">Prerequisites<a aria-label="Copy link to Prerequisites" class="heading-anchor" href="#prerequisites">#</a></h2>
<p>Before starting, make sure you have:</p>
<ol><li>A theStacc project with <strong>Content SEO</strong> enabled and at least one published blog</li><li>Your <strong>Public API Key</strong> (more on where to find it below)</li><li>A static site built with Astro, Next.js, Hugo, Nuxt, or any other static site generator</li><li>Your site deployed on Cloudflare Pages, Vercel, Netlify, or similar</li></ol>
<h2 id="overview">Overview<a aria-label="Copy link to Overview" class="heading-anchor" href="#overview">#</a></h2>
<p>The integration has three parts:</p>
<ol><li><strong>Fetch</strong> — Your site calls the theStacc API at build time to get blog data</li><li><strong>Render</strong> — Your generator turns the API response into static HTML pages</li><li><strong>Rebuild</strong> — A deploy hook triggers a new build whenever you publish or unpublish a post</li></ol>
<p>This guide covers the first two parts. See <a href="/docs/developers/deploy-hooks/">Deploy Hooks</a> for automatic rebuilds.</p>
<h2 id="get-your-api-key">Get your API key<a aria-label="Copy link to Get your API key" class="heading-anchor" href="#get-your-api-key">#</a></h2>
<p>Every project has its own <strong>Public API Key</strong>. To find it, open your project and go to <strong>Content &gt; Settings &gt; Publishing</strong>, then scroll to the <strong>API Access</strong> section.</p>
<p>In that section you'll see:</p>
<ul><li><strong>Public API Key</strong> — your key, starting with <code>pk_live_</code>. Use <strong>Reveal</strong> to show it and <strong>Copy</strong> to copy it to your clipboard.</li><li><strong>Rotate Key</strong> — generates a brand-new key and immediately invalidates the old one. Use this if the key was ever exposed. After rotating, update the key everywhere you've stored it, or your builds will start failing.</li><li><strong>Revoke</strong> — removes the key entirely. Any site using it will stop fetching content until you generate a new one.</li></ul>
<p>If you don't see a key yet, click <strong>Generate API Key</strong> to create one.</p>
<p>Keep your API key private. Store it as an environment variable in your build process — never commit it to your repository.</p>

<p>For your hosting platform, add the same variables in the dashboard so they're available during the build:</p>
<ul><li><strong>Cloudflare Pages</strong> — <strong>Settings &gt; Environment variables</strong></li><li><strong>Vercel</strong> — <strong>Settings &gt; Environment Variables</strong></li><li><strong>Netlify</strong> — <strong>Site configuration &gt; Environment variables</strong></li></ul>
<p>The API base path is <code>https://api.thestacc.com/blog/api/v1/public/blogs</code>. The three endpoints you'll use are:</p>
<ul><li><code>GET /</code> — list published blogs (metadata only by default; add <code>include_content=true</code> for full HTML)</li><li><code>GET /{slug}</code> — one published blog by slug (always includes full HTML content)</li><li><code>GET /sitemap</code> — a lightweight list of slugs and dates for building your sitemap</li></ul>
<p>Full parameters, defaults, and response shapes are in the <a href="/docs/developers/public-blog-api/">Public Blog API reference</a>.</p>

<h2 id="build-time-error-handling">Build-time error handling<a aria-label="Copy link to Build-time error handling" class="heading-anchor" href="#build-time-error-handling">#</a></h2>
<p>Because content is fetched while your site builds, you need a plan for the rare case where the API can't be reached during a build — for example, a transient network blip or a <code>500</code> from the server.</p>
<p><strong>What happens by default.</strong> In each helper above, the list fetch throws an error when the response isn't OK (<code>if (!res.ok) throw new Error(...)</code>). That's intentional: if the blog list can't be fetched, the build <strong>fails loudly and stops</strong>, and your hosting platform keeps the previous successful deploy live. Your visitors keep seeing the last good version of your site — they never see a half-built site with missing posts. This is the safest default.</p>
<p>The single-blog helpers instead return <code>null</code> on a non-OK response so one missing or unpublished post (a <code>404</code>) doesn't take down the whole build — the page-not-found path handles it gracefully.</p>
<p><strong>Add a short retry.</strong> Most build failures are momentary. A small retry-with-backoff around the list fetch makes builds far more resilient:</p>

<p>Don't retry a <code>401</code> (your API key is wrong, missing, or was revoked) or a <code>404</code> — those won't fix themselves. Do retry <code>500</code> and network errors. See the <a href="/docs/developers/public-blog-api#error-handling/">Public Blog API error reference</a> for the full list of status codes.</p>
<p><strong>Don't call the API from the browser.</strong> The Public Blog API is built for build-time fetching, not live client requests — a typical build makes only a handful of calls. Fetching from client-side code would expose your key and add a runtime dependency on the API. Always fetch at build time.</p>
<p><strong>Incremental Static Regeneration (ISR) as an alternative.</strong> If you'd rather your pages refresh on a schedule instead of failing the build, frameworks like Next.js support ISR: serve the last-built version, then quietly re-fetch in the background every N seconds. Set <code>revalidate</code> to a number of seconds (for example, <code>next: { revalidate: 3600 }</code> for hourly) instead of <code>false</code>. With ISR, a momentary API outage means visitors simply keep seeing the previous cached page until the next successful refresh — no broken build, no missing content. The trade-off is that new posts can take up to your revalidate window to appear unless a <a href="/docs/developers/deploy-hooks/">deploy hook</a> also triggers a rebuild. Most sites are happiest with a deploy hook plus the fail-loud default above; ISR is an option if you want time-based refreshes on top.</p>
<h2 id="keeping-your-site-up-to-date-with-deploy-hooks">Keeping your site up to date with Deploy Hooks<a aria-label="Copy link to Keeping your site up to date with Deploy Hooks" class="heading-anchor" href="#keeping-your-site-up-to-date-with-deploy-hooks">#</a></h2>
<p>Fetching at build time means new posts only appear when your site rebuilds. You don't want to trigger that by hand every time. A <strong>deploy hook</strong> does it for you: theStacc sends a request to your hosting platform whenever a post is <strong>published</strong> or <strong>unpublished</strong>, which kicks off a fresh build that pulls in the latest content.</p>
<p>You'll find the <strong>Deploy Hook</strong> field right below your API key, under <strong>Content &gt; Settings &gt; Publishing &gt; API Access</strong>. Paste the deploy hook URL from your Cloudflare Pages, Vercel, or Netlify dashboard, click <strong>Save</strong>, then click <strong>Test</strong> to confirm it triggers a build. The URL must be an HTTPS URL.</p>
<p>For the full walkthrough — getting the URL from each platform, GitHub Pages via GitHub Actions, and troubleshooting — see <a href="/docs/developers/deploy-hooks/">Deploy Hooks</a>.</p>

<p>If your generator already produces a sitemap that includes your blog routes (such as <code>@astrojs/sitemap</code>), you don't need to call this endpoint separately — the blog pages are picked up automatically because they're static routes.</p>

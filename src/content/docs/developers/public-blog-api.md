title: Public Blog API
---

 <p>The Public Blog API lets you fetch your published blog content from theStacc and display it on any website — including static sites hosted on Cloudflare Pages, Vercel, Netlify, or any other platform. No CMS required.</p>
<h2 id="when-to-use-this">When to use this<a aria-label="Copy link to When to use this" class="heading-anchor" href="#when-to-use-this">#</a></h2>
<p>Use the Public Blog API when:</p>
<ul><li>Your site runs on <strong>Cloudflare Pages</strong>, <strong>Vercel</strong>, <strong>Netlify</strong>, or <strong>GitHub Pages</strong></li><li>You don't use a CMS like WordPress, Ghost, or Webflow</li><li>Your site is built with a static site generator (Astro, Next.js, Nuxt, Hugo, Gatsby, 11ty, etc.)</li><li>You want full control over how blog content is rendered on your site</li></ul>
<p>If you use WordPress, Ghost, Webflow, or Zepio, you don't need this — use the <a href="/docs/content-seo/publishing/">direct publishing integrations</a> instead, which push posts straight into your CMS.</p>
<p>This is the <strong>pull</strong> side of theStacc's integrations: your site reads content from us. For an end-to-end walkthrough of wiring it into a specific framework, follow <a href="/docs/developers/static-site-integration/">Static Site Integration</a>.</p>
<h2 id="how-it-works">How it works<a aria-label="Copy link to How it works" class="heading-anchor" href="#how-it-works">#</a></h2>
<p>theStacc generates and stores your blog content. The Public Blog API exposes the <strong>published</strong> posts for one project as JSON. Your static site fetches them at <strong>build time</strong> and generates static HTML pages.</p>

<h2 id="authentication">Authentication<a aria-label="Copy link to Authentication" class="heading-anchor" href="#authentication">#</a></h2>
<p>Every project gets one <strong>Public API Key</strong> for read-only public access. Find it in your project under <strong>Settings &gt; Publishing &gt; API Access</strong>. Click <strong>Generate API Key</strong> if you don't have one yet; <strong>Reveal</strong> and <strong>Copy</strong> to read it.</p>
<p>The key always starts with <code>pk_live_</code>. It is project-scoped: it only ever returns the published blogs that belong to that one project.</p>
<p>Include the key in every request — either as a query parameter or as a Bearer header:</p>

<h3 id="this-is-a-project-key-not-an-agent-key">This is a project key, not an agent key<a aria-label="Copy link to This is a project key, not an agent key" class="heading-anchor" href="#this-is-a-project-key-not-an-agent-key">#</a></h3>
<p>The <code>pk_live_</code> Public API Key is <strong>read-only</strong>. It can list and read your published blogs — nothing more. It cannot create, edit, publish, or delete content, and it can't see drafts or unpublished posts. That's why it's safe to use in a static-site build.</p>
<p>It is a completely different credential from the <strong>agent keys</strong> (which start with <code>sk_agent_</code>) used by the <a href="/docs/developers/agent-keys-mcp/">Model Context Protocol (MCP) server</a>. Agent keys are secret, write-capable (they can generate, edit, and publish blogs), are stored hashed, and live in the <strong>Agent access</strong> section right beside API Access. Never use one in place of the other — and never embed an <code>sk_agent_</code> key in a public build.</p>
<h3 id="keep-your-key-safe">Keep your key safe<a aria-label="Copy link to Keep your key safe" class="heading-anchor" href="#keep-your-key-safe">#</a></h3>
<p>Though the Public API Key is read-only, treat it as a credential. Store it in an environment variable in your build process and <strong>never commit it to your repository</strong>.</p>
<p>If a key is ever exposed, open <strong>Settings &gt; Publishing &gt; API Access</strong> and click <strong>Rotate Key</strong> to issue a new one (the old key stops working immediately), or <strong>Revoke</strong> to disable public access entirely. Any site still using the old key will stop fetching the moment you rotate or revoke, so update your build environment variable at the same time.</p>
<h2 id="endpoints">Endpoints<a aria-label="Copy link to Endpoints" class="heading-anchor" href="#endpoints">#</a></h2>
<p>All endpoints share the base URL <code>https://api.thestacc.com/blog/api/v1/public</code>.</p>
<h3 id="list-published-blogs">List published blogs<a aria-label="Copy link to List published blogs" class="heading-anchor" href="#list-published-blogs">#</a></h3>
<div class="docs-code"><div class="docs-code-bar"><span class="docs-code-lang">Code</span><button aria-label="Copy code" class="docs-code-copy" type="button"><svg class="docs-code-copy-icon-copy" fill="none" height="14" stroke="currentColor" stroke-linecap="round" stroke-linejoin="round" stroke-width="2" viewbox="0 0 24 24" width="14" xmlns="http://www.w3.org/2000/svg"><rect height="14" rx="2" ry="2" width="14" x="8" y="8"></rect><path d="M4 16c-1.1 0-2-.9-2-2V4c0-1.1.9-2 2-2h10c1.1 0 2 .9 2 2"></path></svg><svg class="docs-code-copy-icon-check" fill="none" height="14" stroke="currentColor" stroke-linecap="round" stroke-linejoin="round" stroke-width="2.5" viewbox="0 0 24 24" width="14" xmlns="http://www.w3.org/2000/svg"><path d="M20 6 9 17l-5-5"></path></svg><span class="docs-code-copy-label">Copy</span></button></div><pre class="shiki" tabindex="0"><code><span class="line">GET /blog/api/v1/public/blogs?api_key=YOUR_API_KEY</span></code></pre></div>
<p>Returns blogs with <code>status: published</code>, sorted by <code>published_at</code> (newest first). By default it returns <strong>metadata only</strong> (no body content) for a fast, lightweight listing. Pass <code>include_content=true</code> to get the full HTML body for every post in the page.</p>
<p><strong>Query parameters:</strong></p>
<ul><li><code>api_key</code> (required) — your project's <code>pk_live_</code> key (or pass it via the Authorization header instead)</li><li><code>limit</code> (optional) — number of blogs to return. Default <code>50</code>, minimum <code>1</code>, maximum <code>100</code></li><li><code>offset</code> (optional) — pagination offset. Default <code>0</code>, minimum <code>0</code></li><li><code>category</code> (optional) — return only blogs that include this category name</li><li><code>include_content</code> (optional) — set to <code>true</code> to include the full HTML body of each blog. Default <code>false</code></li></ul>
<p>Use <code>limit</code> and <code>offset</code> together to page through more than 100 posts. The <code>total</code> field in the response is the full count of published blogs (after any <code>category</code> filter), so you can compute how many pages remain.</p>

<h2 id="content-format">Content format<a aria-label="Copy link to Content format" class="heading-anchor" href="#content-format">#</a></h2>
<p>The <code>content</code> field is <strong>HTML</strong> — the same body HTML theStacc generates for your post. It includes:</p>
<ul><li>Headings (<code>h2</code>, <code>h3</code>, <code>h4</code>)</li><li>Paragraphs, lists, and blockquotes</li><li>Links (internal and external)</li><li>Images with alt text</li><li>Inline styles for formatting</li></ul>
<p>You can render it directly with <code>set:html</code> (Astro), <code>dangerouslySetInnerHTML</code> (React/Next.js), or <code>| safe</code> (Hugo). Apply your own CSS to match your site's design.</p>
<h3 id="the-featured-image-is-separate-from-the-body">The featured image is separate from the body<a aria-label="Copy link to The featured image is separate from the body" class="heading-anchor" href="#the-featured-image-is-separate-from-the-body">#</a></h3>
<p><code>featured_image_url</code> is a <strong>standalone field</strong>, not part of the <code>content</code> HTML. theStacc treats your post's hero image (or, if there's no dedicated hero, the first generated image) as a structured field so you can render it however you like — as a hero banner above the article, as a card thumbnail in your listing, or in your social/OG tags.</p>
<p>The body <code>content</code> is the article itself and is <strong>not</strong> prefixed with the featured image, so the hero won't appear twice. In your template, render <code>featured_image_url</code> as the hero and render <code>content</code> as the body. Any other images that belong inside the article stay inside <code>content</code> as normal <code>&lt;img&gt;</code> tags.</p>
<h3 id="cross-post-links-are-resolved-for-you">Cross-post links are resolved for you<a aria-label="Copy link to Cross-post links are resolved for you" class="heading-anchor" href="#cross-post-links-are-resolved-for-you">#</a></h3>
<p>When theStacc writes a post that links to another one of your posts, it stores that link as an internal placeholder (<code>[INTERNAL:other-post-slug]</code>) rather than a hard URL, so the link survives no matter where you publish. The Public API <strong>resolves these placeholders for you on every read</strong> whenever it returns body content (the single-blog endpoint, and the list endpoint when <code>include_content=true</code>):</p>
<ul><li>If the referenced slug matches another <strong>published</strong> post in the same project, the placeholder becomes a real link to that post on your site, using the path convention <code>your-site.com/blog/&lt;slug&gt;/</code>. The base URL comes from the website connected to your project; if no site URL is set, the link is rendered as a root-relative <code>/blog/&lt;slug&gt;/</code> path.</li><li>If the referenced post <strong>isn't published yet</strong> (or the slug doesn't match anything), the link is dropped but the anchor text is kept. You'll never receive a broken link or a raw <code>[INTERNAL:...]</code> placeholder.</li></ul>
<p>Because resolution happens at read time, fixing or publishing the target post is reflected on your next fetch — you don't need to regenerate or republish the post that links to it.</p>
<p>This matches the <code>your-site.com/blog/&lt;slug&gt;/</code> URL pattern used throughout the <a href="/docs/developers/static-site-integration/">Static Site Integration</a> guide. If your blog lives at a different path, you'll want to keep that path as <code>/blog/</code> (or adjust your routing accordingly) so resolved internal links point to the right place.</p>
<h3 id="self-hosting-images">Self-hosting images<a aria-label="Copy link to Self-hosting images" class="heading-anchor" href="#self-hosting-images">#</a></h3>
<p>Images referenced in <code>content</code> and <code>featured_image_url</code> are served from theStacc storage. For best performance and to avoid depending on our storage, many teams download these images during their build and rewrite the URLs to their own CDN. This is optional — the URLs work as-is.</p>
<h2 id="rate-limits">Rate limits<a aria-label="Copy link to Rate limits" class="heading-anchor" href="#rate-limits">#</a></h2>
<p>The Public Blog API is designed for <strong>build-time</strong> fetching, not real-time client requests. A typical static site build makes a handful of requests (a list or sitemap call, plus one per post). Fetch the content during your build, generate static pages, and serve those — don't call the API from client-side browser code on every page view.</p>
<p>If you have a very large catalog, page through the list endpoint with <code>limit</code>/<code>offset</code> rather than requesting everything at once, and use the lightweight <code>sitemap</code> endpoint (which returns just slugs and dates) when you only need URLs.</p>

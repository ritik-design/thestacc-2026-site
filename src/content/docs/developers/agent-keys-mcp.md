---
title: Agent Keys & MCP Server
---

 <p>The theStacc MCP server lets an AI agent drive your blog workflow for you. Instead of clicking through the dashboard, you can ask an agent like Claude Desktop or Cursor to *"review my content plan, generate the next post, and show me the draft."* The agent does the work; you stay in control of what gets published.</p>
<p>This page explains what the MCP server is, how the <strong>agent key</strong> that connects it works, and every tool the agent can use (with its limits). It's written for the person setting this up, technical or not.</p>
<h2 id="what-the-mcp-server-is">What the MCP server is<a aria-label="Copy link to What the MCP server is" class="heading-anchor" href="#what-the-mcp-server-is">#</a></h2>
<p>MCP (Model Context Protocol) is an open standard that lets AI agents use external tools. The <strong>theStacc MCP server</strong> is a small program that gives your agent a set of tools for your blog project: see your plan, generate and edit drafts, swap in your own images, and publish posts you've approved.</p>
<p>A few things make it easy to use:</p>
<ul><li><strong>Nothing to install.</strong> Your agent client launches the server on demand with <code>npx</code>. You don't install anything globally.</li><li><strong>Runs locally.</strong> It runs on your own machine as a Node.js process and talks to theStacc over a secure connection. It needs <strong>Node.js 18 or newer</strong> (<code>npx</code> ships with it).</li><li><strong>Works with any MCP-capable client</strong> — Claude Desktop, Cursor, or your own agent runtime.</li></ul>
<p>The server is published as <code>@thestacc/mcp-server</code>. It's configured with two environment variables:</p>
<ul><li><strong><code>THESTACC_AGENT_KEY</code></strong> (required) — your project agent key, starting with <code>sk_agent_</code>. This is how the server proves it's allowed to act on your project.</li><li><strong><code>THESTACC_API_URL</code></strong> (optional) — the theStacc blog service address. You almost never need to set this; it defaults to <code>https://api.thestacc.com/blog</code>.</li></ul>

In <strong>Cursor</strong>, add the same block to <code>~/.cursor/mcp.json</code> (global) or <code>.cursor/mcp.json</code> (per-project). The shape is identical:</p>

<h2 id="agent-keys">Agent keys<a aria-label="Copy link to Agent keys" class="heading-anchor" href="#agent-keys">#</a></h2>
<p>An <strong>agent key</strong> is a secret that connects one agent to one project. Think of it like a password for your AI assistant — it lets the agent act on your project on your behalf, with editor-level permissions.</p>
<h3 id="how-to-create-one">How to create one<a aria-label="Copy link to How to create one" class="heading-anchor" href="#how-to-create-one">#</a></h3>
<p>Agent keys are managed in the dashboard under <strong>Content SEO &gt; Settings &gt; Publishing &gt; Agent access</strong> (currently labelled <strong>Beta</strong>). Only a <strong>project owner or admin</strong> can create or manage them.</p>
<ol><li>Go to <strong>Settings &gt; Publishing</strong> and find the <strong>Agent access</strong> section.</li><li>Optionally type a <strong>label</strong> (e.g. "My laptop" or "Cursor") so you can tell your keys apart later. Labels can be up to 120 characters.</li><li>Click <strong>Generate agent key</strong>.</li><li>A dialog shows the full key (it starts with <code>sk_agent_</code>) and a ready-to-paste config snippet. <strong>Copy it now.</strong></li><li>Click <strong>I've saved it</strong> to close the dialog.</li></ol>
<h3 id="important-things-to-know">Important things to know<a aria-label="Copy link to Important things to know" class="heading-anchor" href="#important-things-to-know">#</a></h3>
<ul><li><strong>Shown once.</strong> The full secret is displayed exactly one time, at creation. theStacc never stores the secret itself — only a one-way hash — so it can't show it to you again. If you lose it, revoke it and create a new one.</li><li><strong>Project-scoped.</strong> A key works on the one project it was created for, and nothing else. If you have several projects, each needs its own key.</li><li><strong>Prefix for identification.</strong> In the dashboard, each key is shown by its first 16 characters (the prefix) followed by dots, e.g. <code>sk_agent_AbC123••••</code>. That prefix is just for you to recognize the key in the list — the rest stays secret.</li><li><strong>Revocable any time.</strong> Click <strong>Revoke</strong> next to a key. Any agent using it stops working immediately. Revoking can't be undone — create a new key if you need access again.</li><li><strong>Usage tracking.</strong> The list shows each key's label and when it was last used, so you can spot keys you no longer need.</li><li>theStacc shows up to 100 keys per project in the Agent access list. In practice you'll only need one or two.</li></ul>
<h2 id="agent-key-vs-the-public-blog-api-key">Agent key vs. the Public Blog API key<a aria-label="Copy link to Agent key vs. the Public Blog API key" class="heading-anchor" href="#agent-key-vs-the-public-blog-api-key">#</a></h2>
<p>theStacc has two different kinds of project keys. They are not interchangeable:</p>
<div class="table-wrap"><table><thead><tr><th></th><th><strong>Agent key</strong> (<code>sk_agent_...</code>)</th><th><strong>Public Blog API key</strong> (<code>pk_live_...</code>)</th></tr></thead><tbody><tr><td><strong>Purpose</strong></td><td>Let an AI agent plan, generate, edit, and publish</td><td>Read your published posts to show on a website</td></tr><tr><td><strong>Access</strong></td><td>Read <strong>and write</strong> (editor-level, approval-gated publish)</td><td><strong>Read-only</strong></td></tr><tr><td><strong>Where it goes</strong></td><td>In your private agent client config</td><td>Embedded in a website or static-site build</td></tr><tr><td><strong>Created by</strong></td><td>Project owner/admin, under <strong>Agent access</strong></td><td>Project owner/admin, under <strong>API Access</strong></td></tr><tr><td><strong>Shown</strong></td><td>Once, at creation</td><td>Retrievable in settings (it's meant to be embedded)</td></tr></tbody></table></div>
<p>In short: the <strong>Public Blog API key is safe to put in a website</strong> because it can only read published content. The <strong>agent key is a powerful secret</strong> — it can change and publish your content — so it must stay private. Never swap one for the other.</p>
<p>For the read-only key and the public read API, see <a href="/docs/developers/public-blog-api/">Public Blog API</a>.</p>
<h2 id="tool-catalog">Tool catalog<a aria-label="Copy link to Tool catalog" class="heading-anchor" href="#tool-catalog">#</a></h2>
<p>Once connected, your agent has 12 tools. You don't call these directly — you ask in plain language and the agent picks the right tool. Here's what each one does so you know what's possible.</p>
<h3 id="see-your-project-and-plan">See your project and plan<a aria-label="Copy link to See your project and plan" class="heading-anchor" href="#see-your-project-and-plan">#</a></h3>
<ul><li><strong><code>get_project_context</code></strong> — Returns which project this key is scoped to (id, account, name). A good sanity check that the key is wired up correctly.</li><li><strong><code>review_content_plan</code></strong> — Shows your current content plan: the list of planned and draft blogs (id, title, status, scheduled date) plus plan metadata. This is the natural place to start a session.</li></ul>
<h3 id="create-content">Create content<a aria-label="Copy link to Create content" class="heading-anchor" href="#create-content">#</a></h3>
<ul><li><strong><code>generate_blog</code></strong> — Starts generating a blog that's already in your plan. It returns immediately with a <code>generating</code> status; the agent polls until it's done. You can pass optional extra steering of up to <strong>2,000 characters</strong>.</li><li><strong><code>generate_more</code></strong> — Adds new planned blogs to your project. The agent adds them in small batches (1 to 10 at a time); they land in an *awaiting date* state, and a human picks publish dates in the dashboard before they can be generated. (The underlying service won't accept more than <strong>200</strong> new blogs in a single request.)</li><li><strong><code>draft_from_abstract</code></strong> — Originates a brand-new blog from your own pointers instead of the auto-plan. You give it a short brief — between <strong>10 and 4,000 characters</strong> — and it starts generating. You can optionally pass a blog type (e.g. <code>how_to</code>).</li><li><strong><code>generation_status</code></strong> — Polls progress for a blog that's generating: status, word count, and how many images are done.</li></ul>
<h3 id="refine-a-draft">Refine a draft<a aria-label="Copy link to Refine a draft" class="heading-anchor" href="#refine-a-draft">#</a></h3>
<ul><li><strong><code>get_blog</code></strong> — Fetches one draft in full: content, title, meta, and status.</li><li><strong><code>edit_blog</code></strong> — Edits a draft's text or metadata (title, content, excerpt, meta title, meta description). Note: editing the content of an already-reviewed blog moves it back to a *pending review* state, so a human re-checks the change.</li><li><strong><code>regenerate_section</code></strong> — Rewrites one section of a draft. Tone and length are 0–100 sliders (0 = more formal/shorter, 100 = more casual/longer). The section can be up to <strong>100,000 characters</strong> of HTML, and you can add a custom prompt of up to <strong>1,000 characters</strong>.</li><li><strong><code>regenerate_image</code></strong> — Regenerates one image in a draft. Style is a 0–100 slider (0 = modern, 50 = abstract, 100 = realistic), with an optional custom prompt up to <strong>1,000 characters</strong>.</li><li><strong><code>upload_image</code></strong> — Drops *your own* image (e.g. a product screenshot) into a blog as real media instead of an AI image. It reads a local file and, by default, replaces the first image in the post. Other placements: <code>append</code> adds it at the end, and <code>none</code> only stores it and returns an image snippet to place yourself with <code>edit_blog</code>. Uploaded images can be up to <strong>15 MB</strong>.</li></ul>
<h3 id="publish">Publish<a aria-label="Copy link to Publish" class="heading-anchor" href="#publish">#</a></h3>
<ul><li><strong><code>publish_blog</code></strong> — Publishes a blog to your connected site. This only works on a blog you've <strong>approved</strong> (see below).</li></ul>
<h2 id="publishing-is-human-gated">Publishing is human-gated<a aria-label="Copy link to Publishing is human-gated" class="heading-anchor" href="#publishing-is-human-gated">#</a></h2>
<p>This is the most important rule, and it's enforced by the server itself:</p>
<p><strong><code>publish_blog</code> only works on a blog that a human has set to *Approved* in the dashboard.</strong> The agent can draft, edit, and prepare a post, but it <strong>cannot approve its own work</strong> — only you can click <strong>Approve</strong>. This keeps a person in the loop before anything goes live.</p>
<ul><li>If you ask the agent to publish a blog that isn't approved, the publish fails with a <strong><code>not_approved</code></strong> error (HTTP <strong>409</strong>), telling you to approve it first.</li><li>If a post is held back by a <strong>compliance check</strong>, publishing comes back as unsuccessful and the agent surfaces the reason. The agent <strong>cannot override a compliance hold</strong>.</li></ul>
<p>A natural workflow: ask the agent to *"review the plan, generate the next post, and show me the draft."* Edit with the agent until you're happy. Then <strong>you</strong> click <strong>Approve</strong> in the dashboard, and ask it to *"publish the approved post."*</p>
<p>To understand the stages a blog moves through — generated, pending review, approved, published — see the blog statuses in the <a href="/docs/content-seo/overview/">Content SEO overview</a> and the approval modes in <a href="/docs/content-seo/publishing/">Publishing</a>.</p>
<h2 id="status-error-codes">Status &amp; error codes<a aria-label="Copy link to Status &amp; error codes" class="heading-anchor" href="#status-error-codes">#</a></h2>
<p>When something goes wrong, the agent receives a clear code. The common ones:</p>
<ul><li><strong>401 — invalid or revoked key.</strong> The agent key is wrong, missing, or has been revoked. Check <code>THESTACC_AGENT_KEY</code>, or generate a new key.</li><li><strong>403 — wrong project or not allowed.</strong> The key isn't authorized for the project or action you asked for.</li><li><strong>404 — not found.</strong> The blog id doesn't exist (or isn't in this project).</li><li><strong>409 — not approved.</strong> You tried to publish a blog that hasn't been approved by a human yet. Approve it in the dashboard first.</li><li><strong>413 — too large.</strong> An upload exceeded a size limit (e.g. an image over 15 MB).</li><li><strong>422 — validation error.</strong> An input was out of range or malformed (for example, an abstract shorter than 10 characters).</li><li><strong>5xx — temporary server error.</strong> A transient hiccup on our side. These are safe to <strong>retry</strong> after a moment.</li></ul>
<h2 id="security">Security<a aria-label="Copy link to Security" class="heading-anchor" href="#security">#</a></h2>
<p>Agent keys are powerful, so handle them like passwords:</p>
<ul><li><strong>Stored safely.</strong> theStacc never keeps the plaintext key. Only a SHA-256 hash is stored, so even we can't read your key back to you — which is why it's shown only once.</li><li><strong>Keep it private.</strong> Never embed an agent key in a website, a public repo, a shared document, or client-side code. It belongs in your private agent client config. If a key leaks, <strong>revoke it immediately</strong> and create a new one.</li><li><strong>Scoped and revocable.</strong> A key only ever touches its one project, and you can shut it off instantly from <strong>Agent access</strong>.</li><li><strong>Safe publishing targets.</strong> When the agent publishes to a connected site, theStacc's standard safeguards still apply — including protections that block publishing to internal or private network addresses (SSRF guards). The agent can't reach anything it shouldn't.</li></ul>

title: Webhook Reference
---

 <p>This is the <strong>end-to-end reference</strong> for theStacc's Custom Webhook integration. It covers everything a developer needs to ship a production receiver: how authentication works, every event we send, every field we put in the payload, what to return, how retries and idempotency behave, and ready-to-paste receiver and database-schema examples.</p>
<blockquote><p><strong>Quick orientation:</strong> if you're configuring this from theStacc's dashboard, start at <a href="/docs/content-seo/connect-platforms/">Connect Platforms</a> and choose Custom Webhook. This page is for the developer who needs to build (or audit) the receiving endpoint. For what happens when a publish fails and how to recover, see <a href="/docs/developers/publishing-errors-retries/">Publishing Errors &amp; Retries</a>.</p></blockquote>
<h2 id="when-to-use-a-webhook">When to use a webhook<a aria-label="Copy link to When to use a webhook" class="heading-anchor" href="#when-to-use-a-webhook">#</a></h2>
<p>Choose the Custom Webhook integration when:</p>
<ul><li>Your site is built on a <strong>custom stack</strong> — Next.js, Astro, Remix, SvelteKit, Rails, Django, Laravel, etc. (anything that isn't one of our direct integrations).</li><li>You want <strong>full control</strong> over what happens when a blog publishes — write to your own database, trigger a rebuild, push to a CDN, fan out to other systems.</li><li>You already have a CMS but want <strong>theStacc to write into it via your own API</strong>, not theStacc's direct integrations.</li></ul>
<p>If you use WordPress, Webflow, Ghost, Shopify, or Zepio, use those integrations instead — they handle authentication, image uploads, and field mapping for you. See <a href="/docs/content-seo/connect-platforms/">Connect Platforms</a> for the full list.</p>
<h2 id="how-the-handshake-works">How the handshake works<a aria-label="Copy link to How the handshake works" class="heading-anchor" href="#how-the-handshake-works">#</a></h2>
<p>theStacc uses <strong>stateless per-request authentication</strong> via a shared secret — the same model as Stripe, GitHub, and Slack webhooks. There is <strong>no OAuth dance, no token exchange, no TLS-mutual-auth</strong>. The "handshake" happens once during setup: both sides record the same secret, then every subsequent request is independently signed and verified.</p>
<h3 id="step-1-one-time-setup">Step 1 — One-time setup<a aria-label="Copy link to Step 1 — One-time setup" class="heading-anchor" href="#step-1-one-time-setup">#</a></h3>
<p>The developer pastes a Webhook URL and Secret Key into the <strong>Content SEO &gt; Settings &gt; Publishing</strong> dialog when adding a Custom Webhook. theStacc stores the secret; the developer keeps a copy in an env var on their server. After this, both sides hold the same secret.</p>
<figure class="docs-figure"><img alt="Webhook one-time setup — developer pastes URL + Secret Key into theStacc dashboard, theStacc stores it" loading="lazy" src="/images/docs/webhook-setup.webp"><figcaption>Webhook one-time setup — developer pastes URL + Secret Key into theStacc dashboard, theStacc stores it</figcaption></img></figure>
<h3 id="step-2-per-request-every-blog-publish-sync-unpublish-delete">Step 2 — Per request (every blog publish, sync, unpublish, delete)<a aria-label="Copy link to Step 2 — Per request (every blog publish, sync, unpublish, delete)" class="heading-anchor" href="#step-2-per-request-every-blog-publish-sync-unpublish-delete">#</a></h3>
<p>theStacc serializes the payload to JSON, computes <code>sig = HMAC-SHA256(secret, body)</code>, and POSTs the body to your webhook URL with the signature in the <code>X-Webhook-Signature</code> header. Your receiver reads the <strong>raw body bytes</strong> (not the parsed JSON), recomputes the same HMAC with its own copy of the secret, and compares. If the two signatures match, the request is authentic — process it and respond <code>200</code> with the live URL and your CMS id. If they don't, return <code>401</code>.</p>
<figure class="docs-figure"><img alt="Webhook per-request signing flow — theStacc signs body with HMAC-SHA256 and sends X-Webhook-Signature header; receiver verifies and returns 200 OK with url and id" loading="lazy" src="/images/docs/webhook-signing.webp"/><figcaption>Webhook per-request signing flow — theStacc signs body with HMAC-SHA256 and sends X-Webhook-Signature header; receiver verifies and returns 200 OK with url and id</figcaption></figure>
<p>Both sides compute the same signature independently. If they match, the receiver knows the request <strong>really came from theStacc</strong> and <strong>wasn't modified in transit</strong>. That's the entire authentication mechanism.</p>
<blockquote><p><strong>The shared secret is the entire defense.</strong> Anyone who guesses your webhook URL but doesn't know the secret cannot forge a request. Treat it like a database password — store it in env vars, rotate it if leaked, and never commit it to git.</p></blockquote>
<h2 id="30-minute-production-setup">30-minute production setup<a aria-label="Copy link to 30-minute production setup" class="heading-anchor" href="#30-minute-production-setup">#</a></h2>
<p>A field-tested path from "we picked Webhook" to "first blog publishes live":</p>
<ol><li><strong>Generate a secret</strong> — <code>openssl rand -hex 32</code> (the secret must be <strong>16+ characters</strong>, which theStacc enforces at save time).</li><li><strong>Add an env var</strong> to your hosting platform: <code>STACC_WEBHOOK_SECRET=&lt;the secret&gt;</code>.</li><li><strong>Add a database table</strong> for incoming blogs (schema templates below).</li><li><strong>Add a receiver route</strong> at, for example, <code>POST /api/stacc-webhook</code> (examples in six languages below).</li><li><strong>Deploy.</strong> Note the public HTTPS URL.</li><li><strong>In theStacc:</strong> go to <strong>Content SEO &gt; Settings &gt; Publishing</strong>, add a Custom Webhook, enter the URL and secret, then click <strong>Test Connection</strong>. A green check means it's working.</li><li><strong>Click Sample Payload</strong> to validate your full publish handler with a fake blog (<code>blog_id</code> will start with <code>preview-</code>).</li><li><strong>Approve a blog and publish.</strong> The first real blog should land in your database within a few seconds.</li></ol>
<p>If anything fails, the <a href="#troubleshooting">Troubleshooting</a> section below covers every error we've seen, and <a href="/docs/developers/publishing-errors-retries/">Publishing Errors &amp; Retries</a> explains how theStacc surfaces and recovers from a failed publish.</p>
<h2 id="authentication-hmac-sha256-signing">Authentication: HMAC-SHA256 signing<a aria-label="Copy link to Authentication: HMAC-SHA256 signing" class="heading-anchor" href="#authentication-hmac-sha256-signing">#</a></h2>
<p>When you configure a webhook secret, every request includes a signature in the <code>X-Webhook-Signature</code> header.</p>

<p><strong>Critical detail:</strong> the signature is always computed over the <strong>exact bytes theStacc puts on the wire</strong>, and theStacc POSTs those same bytes as the body. Your receiver MUST hash the <strong>raw body bytes it received</strong>, not parse-and-re-serialize the JSON. If you re-serialize, whitespace and key order can shift and the signature won't match. This is the single most common integration mistake.</p>
<blockquote><p><strong>Why "hash the raw bytes" matters even more than it looks.</strong> theStacc has two internal publish paths — a manual path (you click Publish) and an autopilot path (theStacc publishes on a schedule). The two paths serialize the JSON slightly differently internally (one compact, one with spacing), but <strong>each path always signs the exact bytes it sends</strong>. A receiver that hashes the raw body works identically on both paths. A receiver that re-serializes the parsed JSON can pass on one path and fail on the other. Hash the raw bytes and you never have to think about this.</p></blockquote>
<h3 id="properties-of-the-signature">Properties of the signature<a aria-label="Copy link to Properties of the signature" class="heading-anchor" href="#properties-of-the-signature">#</a></h3>
<div class="table-wrap"><table><thead><tr><th>Property</th><th>Behavior</th></tr></thead><tbody><tr><td>Deterministic</td><td>Same secret + same body bytes always produce the same signature</td></tr><tr><td>Body-bound</td><td>Even one byte different in the body produces a completely different signature</td></tr><tr><td>Length</td><td>Always 64 hex characters (256 bits)</td></tr><tr><td>Format</td><td>Lowercase hex, no <code>sha256=</code> prefix on the <code>X-Webhook-Signature</code> header</td></tr></tbody></table></div>
<h3 id="legacy-x-fairview-signature-compatibility-header">Legacy <code>X-Fairview-Signature</code> compatibility header<a aria-label="Copy link to Legacy X-Fairview-Signature compatibility header" class="heading-anchor" href="#legacy-x-fairview-signature-compatibility-header">#</a></h3>
<p>On autopilot (scheduled) publishes, theStacc sends a <strong>second</strong> signature header alongside <code>X-Webhook-Signature</code>:</p>

<p><code>X-Fairview-Signature</code> is a <strong>legacy compatibility header</strong> retained for receivers built before the standard <code>X-Webhook-Signature</code> header existed. Both headers carry the <strong>same</strong> HMAC-SHA256 of the same body bytes, so you can verify against either one. New receivers should read <code>X-Webhook-Signature</code> and ignore <code>X-Fairview-Signature</code>. Note that <code>X-Fairview-Signature</code> is <code>sha256=</code>-prefixed, so strip the <code>sha256=</code> prefix before comparing if you choose to use it.</p>

<h2 id="all-events-thestacc-emits">All events theStacc emits<a aria-label="Copy link to All events theStacc emits" class="heading-anchor" href="#all-events-thestacc-emits">#</a></h2>
<p>Five event types. <strong>Branch on the <code>event</code> field at the top of your handler.</strong></p>
<div class="table-wrap"><table><thead><tr><th>Event</th><th>When it fires</th><th>Body shape</th></tr></thead><tbody><tr><td><code>test.ping</code></td><td>User clicks Test Connection</td><td><code>{ event, message, timestamp }</code></td></tr><tr><td><code>blog.published</code></td><td>First successful publish of a blog</td><td>Full blog payload (see below)</td></tr><tr><td><code>blog.updated</code></td><td>Re-publish or content sync of an existing blog</td><td>Full blog payload — same shape as <code>blog.published</code></td></tr><tr><td><code>blog.unpublished</code></td><td>User unpublishes from theStacc</td><td><code>{ event, blog_id, title }</code></td></tr><tr><td><code>blog.deleted</code></td><td>User deletes a published blog</td><td><code>{ event, blog_id, title }</code></td></tr></tbody></table></div>
<p>Whether theStacc sends <code>blog.published</code> or <code>blog.updated</code> is decided by whether your receiver has published this blog before: the first successful publish fires <code>blog.published</code>; any subsequent re-publish or content sync of the same blog fires <code>blog.updated</code>. <strong>Treat both the same way</strong> — upsert on <code>blog_id</code>.</p>
<p>A note on <code>preview-</code> prefixed <code>blog_id</code>s: when the user clicks <strong>Sample Payload</strong> in the dashboard, theStacc fires a real-shaped <code>blog.published</code> event but with <code>blog_id: "preview-00000000-0000-0000-0000-000000000000"</code>. Your handler should detect the prefix and skip the actual database write so test runs don't pollute production data.</p>

<h2 id="idempotency-and-retries">Idempotency and retries<a aria-label="Copy link to Idempotency and retries" class="heading-anchor" href="#idempotency-and-retries">#</a></h2>
<p>How retries behave depends on which path published the blog. Both paths fire the same events at the same URL — the difference is whether theStacc retries automatically.</p>
<h3 id="manual-publish-you-click-publish">Manual publish (you click Publish)<a aria-label="Copy link to Manual publish (you click Publish)" class="heading-anchor" href="#manual-publish-you-click-publish">#</a></h3>
<p>A manual publish is <strong>single-shot</strong>. If your endpoint times out (the request budget is ~30 seconds) or returns a non-2xx status, the publish fails immediately and the user sees a "Failed to publish" notice. The user clicks <strong>Republish</strong> to try again. theStacc does not auto-retry a manual publish.</p>
<h3 id="autopilot-publish-thestacc-publishes-on-a-schedule">Autopilot publish (theStacc publishes on a schedule)<a aria-label="Copy link to Autopilot publish (theStacc publishes on a schedule)" class="heading-anchor" href="#autopilot-publish-thestacc-publishes-on-a-schedule">#</a></h3>
<p>Autopilot publishes <strong>retry automatically</strong> on transient failures (a timeout, a network error, or a <code>5xx</code> from your receiver). theStacc makes up to <strong>5 attempts</strong> with exponential backoff — roughly <strong>60s, then 120s, 240s, 480s, and 960s</strong>, a full retry window of about <strong>30 minutes</strong>. After 5 failed attempts the publish is marked failed. A <code>4xx</code> from your receiver is treated as permanent and is <strong>not</strong> retried (fix your receiver and re-publish). For the full failure-and-recovery story, see <a href="/docs/developers/publishing-errors-retries/">Publishing Errors &amp; Retries</a>.</p>
<h3 id="why-your-receiver-must-be-idempotent">Why your receiver must be idempotent<a aria-label="Copy link to Why your receiver must be idempotent" class="heading-anchor" href="#why-your-receiver-must-be-idempotent">#</a></h3>
<p>Because autopilot retries, the same blog can arrive at your endpoint <strong>more than once</strong>. The classic case: theStacc POSTs, your CMS creates the post, but the response packet is lost in transit (a network blip), so theStacc sees a timeout and retries — and a naive receiver creates a <strong>duplicate post</strong>.</p>
<p>Two fields on autopilot payloads exist specifically to defend against this:</p>
<ul><li><strong><code>idempotency_key</code></strong> — a string of the form <code>"{blog_id}:{lifecycle}"</code> that is <strong>stable across every retry of a single publish</strong> but <strong>different across separate publish intents</strong> (a fresh publish months later gets a different key). Persist the keys you've already processed and <strong>short-circuit on a repeat</strong> — that makes your receiver safe even if the same publish is delivered twice.</li><li><strong><code>publish_attempt</code></strong> — an integer that is <code>1</code> on the first attempt and increments (<code>2</code>, <code>3</code>, ...) on each retry. It's primarily for your logs and observability: <code>publish_attempt &gt; 1</code> tells you theStacc is retrying after a transient hiccup.</li></ul>
<p>Even without those fields, the simplest and most robust defense is to <strong>UPSERT on <code>blog_id</code></strong> so duplicate deliveries (or a user clicking Republish twice) converge on the same row instead of creating two posts.</p>
<h3 id="how-thestacc-itself-avoids-duplicates-direct-cms-integrations">How theStacc itself avoids duplicates (direct CMS integrations)<a aria-label="Copy link to How theStacc itself avoids duplicates (direct CMS integrations)" class="heading-anchor" href="#how-thestacc-itself-avoids-duplicates-direct-cms-integrations">#</a></h3>
<p>For theStacc's own direct integrations (WordPress, Webflow, Ghost, Shopify), before re-POSTing on a retry theStacc looks up the target CMS <strong>by slug</strong> and treats an existing post as "already published by us" only if it matches a <strong>fingerprint</strong>: the same title <strong>and</strong> a creation timestamp <strong>within a ~35-minute window</strong> (which comfortably covers the ~30-minute retry sequence plus headroom). If that fingerprint matches, theStacc reuses the existing post instead of creating a duplicate. You don't need to implement this for a webhook receiver — <code>idempotency_key</code> plus an UPSERT on <code>blog_id</code> is simpler and just as safe — but the <code>slug</code> field plus the ~35-minute window is the same idea you can mirror if your CMS lacks a stable id.</p>

<p>If your CMS write involves image processing, search re-indexing, or CDN purging, run those in a background job <strong>after</strong> responding <code>2xx</code>. A slow synchronous write risks timing out theStacc's request.</p>

<h2 id="self-hosting-images-optional">Self-hosting images (optional)<a aria-label="Copy link to Self-hosting images (optional)" class="heading-anchor" href="#self-hosting-images-optional">#</a></h2>

<h3 id="notes">Notes<a aria-label="Copy link to Notes" class="heading-anchor" href="#notes">#</a></h3>
<ul><li><strong>Featured image is NOT in <code>images[]</code>.</strong> It's a separate top-level field (<code>featured_image_url</code>). theStacc strips the inline hero from <code>content</code> before sending so the page doesn't render the same image twice.</li><li><strong>Duplicate URLs are deduped.</strong> If the same body image appears twice, the array lists it once. Your <code>content.replaceAll(...)</code> swaps every occurrence in one pass anyway.</li><li><strong><code>alt</code> may be empty.</strong> Decorative images and images the generator didn't caption ship with <code>alt: ""</code>. Pass it through verbatim to preserve accessibility on your re-hosted copy.</li><li><strong>Backward compatible.</strong> Receivers that ignore <code>images</code> continue working unchanged — every other field is identical to before this field was added.</li><li><strong>The URL is the stable identifier.</strong> theStacc does not inject any per-image marker attribute into the <code>&lt;img&gt;</code> tags. URLs in <code>images[]</code> mirror <code>content</code> verbatim, so a one-to-one find/replace works without any normalization step.</li></ul>
<h3 id="migrating-blogs-already-published-with-thestacc-s-urls">Migrating blogs already published with theStacc's URLs<a aria-label="Copy link to Migrating blogs already published with theStacc's URLs" class="heading-anchor" href="#migrating-blogs-already-published-with-thestacc-s-urls">#</a></h3>
<p>If you have blogs already in your CMS with theStacc CDN URLs and you want to switch them to self-hosted images:</p>
<ol><li>Open each blog in theStacc and click <strong>Publish</strong> to re-publish.</li><li>The webhook fires as <code>blog.updated</code> with the <code>images</code> array.</li><li>Your receiver runs the mirror loop above and updates the post (same <code>blog_id</code> means the same CMS record).</li><li>The CMS post now stores your URLs instead of theStacc's.</li></ol>
<p>There's no bulk-republish API — for most accounts this is a one-shot migration of a handful of posts.</p>

<h2 id="security-best-practices">Security best practices<a aria-label="Copy link to Security best practices" class="heading-anchor" href="#security-best-practices">#</a></h2>
<ol><li><strong>Always set a webhook secret.</strong> Without one, anyone who guesses your URL can post fake blogs.</li><li><strong>Use <code>crypto.timingSafeEqual</code> / <code>hmac.compare_digest</code></strong> when comparing signatures — plain <code>===</code> is timing-attack vulnerable.</li><li><strong>Hash the raw body, not the parsed JSON.</strong> Re-serialization changes whitespace and breaks signatures.</li><li><strong>Store the secret in env vars only.</strong> Never commit it to git, never include it in client-side code.</li><li><strong>Rotate the secret on suspected leak.</strong> Generate a new one, update both your env var and theStacc's integration settings, then redeploy. theStacc starts signing with the new secret on the next request.</li><li><strong>Run your receiver behind HTTPS-only.</strong> Add HSTS if you control the domain.</li><li><strong>Rate-limit the receiver.</strong> Even with HMAC, a flood of unauthenticated requests can pressure your endpoint while you reject them. Cloudflare or your hosting platform usually handles this.</li></ol>
<h2 id="faq">FAQ<a aria-label="Copy link to FAQ" class="heading-anchor" href="#faq">#</a></h2>
<p><strong>How does the handshake work?</strong></p>
<p>There's no traditional handshake — it's stateless per-request authentication. The "handshake" is just storing a shared secret on both sides, once, during setup. Every subsequent request is independently authenticated via HMAC.</p>
<p><strong>What identifies a request as coming from theStacc?</strong></p>
<p>The <code>X-Webhook-Signature</code> header verified against your secret. Nothing else is reliable — anyone can fake the body, the User-Agent, or the source IP.</p>
<p><strong>What's the <code>X-Fairview-Signature</code> header for?</strong></p>
<p>It's a legacy compatibility header sent on autopilot publishes, carrying the same HMAC as <code>X-Webhook-Signature</code> but <code>sha256=</code>-prefixed. New receivers should use <code>X-Webhook-Signature</code>. See <a href="#legacy-x-fairview-signature-compatibility-header">Legacy <code>X-Fairview-Signature</code> compatibility header</a>.</p>
<p><strong>Why did the same blog arrive twice?</strong></p>
<p>Autopilot publishes retry automatically on transient failures, so a lost response packet can cause a re-delivery. Dedup on <code>idempotency_key</code> (autopilot payloads include it) or UPSERT on <code>blog_id</code>. See <a href="#idempotency-and-retries">Idempotency &amp; retries</a>.</p>
<p><strong>Can I override the <code>Content-Type</code> or signature headers with a custom header?</strong></p>
<p>No. <code>Content-Type</code> is always <code>application/json</code>, and the signature headers are always set by theStacc. Custom headers with those names are ignored.</p>
<p><strong>What's the maximum title / slug / content length?</strong></p>
<p>theStacc doesn't enforce hard length limits at the API level — generated content stays well within reasonable bounds. If your CMS or DB needs hard caps, the <a href="#field-reference">Field reference</a> table lists safe defaults.</p>
<p><strong>What if my receiver is slow or down when a publish happens?</strong></p>
<p>On a manual publish, that publish fails and the user clicks Republish. On an autopilot publish, theStacc retries automatically (up to 5 attempts over ~30 minutes) before marking it failed. See <a href="/docs/developers/publishing-errors-retries/">Publishing Errors &amp; Retries</a>.</p>
<p><strong>Can I get historical blogs through the webhook?</strong></p>
<p>No — webhooks are forward-only. For backfills, use the <a href="/docs/developers/public-blog-api/">Public Blog API</a>, which lets you fetch all your published blogs.</p>
<p><strong>Do I need to handle <code>test.ping</code> separately?</strong></p>
<p>Yes — it has no blog fields. If your handler tries to read <code>title</code> or <code>content</code> from a <code>test.ping</code> event, it'll error and Test Connection will fail. Branch on <code>event</code> first.</p>
<p><strong>Can theStacc post to multiple webhook URLs?</strong></p>
<p>Yes — set up multiple Custom Webhook integrations on the same project. Each gets its own URL, secret, and headers, and theStacc fires every published blog at every active integration.</p>
<p><strong>Does an agent or the MCP server publishing my blog use a different webhook?</strong></p>
<p>No. Whether a teammate clicks Publish or an automation publishes through the MCP server, the publish flows through the same path and fires the same webhook events at the same URL. See <a href="/docs/developers/agent-keys-mcp/">Agent Keys &amp; MCP</a>.</p>
<p><strong>What happens if I delete a blog in theStacc?</strong></p>
<p>We send <code>event: "blog.deleted"</code> with the <code>blog_id</code> and <code>title</code>. Your handler should remove or hide the post. If your receiver fails, the blog stays deleted in theStacc but lingers on your site — periodic reconciliation via the <a href="/docs/developers/public-blog-api/">Public Blog API</a> is a defensive option.</p>
